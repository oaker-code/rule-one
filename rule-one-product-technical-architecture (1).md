# Rule One 产品技术架构文档

## 1. 文档信息

- **产品名称**：Rule One
- **文档类型**：产品技术架构
- **版本**：v1.0
- **定位**：面向全职炒股用户的复盘智能体应用
- **核心目标**：通过结构化复盘，帮助用户在收盘后获得秩序感，而非提供投资建议

---

## 2. 产品概述

### 2.1 产品定义

Rule One 是一个聚焦于“收盘后复盘”的 AI 智能体应用。

它不负责：
- 个股推荐
- 买卖建议
- 行情预测
- 收益承诺
- 盘中带单

它只负责一件事：

> 帮助用户把混乱的一天整理成一条明天可执行的规则。

### 2.2 核心用户价值

Rule One 为用户提供的不是信息优势，而是复盘时的秩序感。

用户在使用后应获得三种明确感受：
1. 我看清了今天发生了什么。
2. 我知道自己哪里偏了。
3. 我知道明天只改哪一件事。

### 2.3 核心产物

每次复盘输出固定为三项：
- **今日关键偏差**
- **今日做对的一点**
- **明日一条规则（Rule One）**

---

## 3. 产品目标与边界

### 3.1 产品目标

Rule One 的目标是构建一个低刺激、非羞辱、强结构化的复盘系统。

具体目标：
- 降低用户复盘门槛
- 减少复盘中的自责和混乱
- 强化用户对自身行为模式的理解
- 形成每日一条可执行规则
- 积累长期行为模式数据，帮助用户建立行为纪律

### 3.2 产品边界

Rule One 明确不进入以下范围：
- 投资顾问服务
- 个股分析推荐
- 交易策略指令
- 收益率预测
- 高情绪粘性陪伴关系
- 社交群体煽动

### 3.3 设计原则

1. **一次只做一件事**：每天只完成一次结构化复盘。
2. **一次只指出一个主问题**：避免一次性输出多个偏差。
3. **一次只给一条规则**：保持 Rule One 的产品核心。
4. **只评论行为，不评论市场**：聚焦用户行为系统。
5. **低刺激、去表演化**：不做排名、炫耀、连胜强化。
6. **非羞辱式输出**：评价行为，不评价人格。

## 5. 核心业务流程

### 5.1 日常主流程

用户每日复盘主链路如下：

```text
收盘进入
  → 情绪选择
  → 今日事实输入
  → AI结构化整理
  → AI识别主偏差
  → AI生成 Rule One
  → 用户确认并保存
  → 写入历史记录
```

### 5.2 流程目标

该流程需要满足以下要求：
- 3 至 7 分钟完成
- 低认知负担
- 强结构化
- 输出固定且可回顾
- 不扩展到行情分析

### 5.3 交互步骤说明

#### Step 1：情绪落地
用户先选择当前情绪状态：
- 懊悔
- 焦虑
- 不服气
- 麻木
- 兴奋
- 平静

作用：让用户先从情绪中抽离出来，进入复盘模式。

#### Step 2：事实回放
用户输入今日事件，可采用：
- 自由文本
- 语音输入
- 固定问题表单

建议固定问题：
- 今天做了几笔
- 最大仓位是多少
- 最大回撤是多少
- 最偏离计划的一笔是哪笔
- 今天最强烈的情绪是什么

#### Step 3：结构化整理
AI 自动整理为四个维度：
- 事实
- 判断
- 情绪
- 动作

#### Step 4：主偏差识别
AI 从所有行为中只识别一个核心偏差，例如：
- 追涨型冲动
- 报复性交易
- 临时加仓
- 死扛不认错
- 盈利后膨胀
- 高频无效操作

#### Step 5：Rule One 输出
AI 最终输出：
- 今日关键偏差
- 今日做对的一点
- 明日唯一规则

#### Step 6：结果存档
生成 Reflection Card，并写入历史记录。

## 7. AI 智能体架构

Rule One 的 AI 逻辑不建议设计为一个单一万能助手，而应采用多角色流水线结构。

### 7.1 智能体角色划分

#### 7.1.1 Emotion Catcher
职责：识别用户当前情绪状态。

输入：
- 用户选择的情绪标签
- 用户自由输入文本

输出：
- 当前主情绪标签
- 建议交互语气

作用：
- 确定后续话术的节奏和强度
- 在焦虑、懊悔、兴奋等状态下采用不同表达策略

#### 7.1.2 Review Structurer
职责：将原始输入整理成结构化复盘内容。

输出格式：
- 事实
- 判断
- 情绪
- 动作

作用：
- 生成秩序感
- 降低用户主观混乱感
- 为偏差识别提供结构化基础

#### 7.1.3 Bias Detector
职责：识别一个主偏差标签。

输入：
- 结构化复盘结果
- 用户历史行为画像

输出：
- 当日主偏差
- 偏差解释

规则：
- 一次只允许输出一个主偏差
- 不应输出多个并列偏差

#### 7.1.4 Self-Respect Guard
职责：把用户的自我攻击语言转化为可修复的行为表达。

示例：
- 用户原话：我真是废物，又管不住手
- 系统内部改写：用户在连续亏损后出现羞耻与失控感，应采用非人格化反馈

作用：
- 降低羞耻感
- 防止复盘变成自我攻击
- 增强用户持续使用意愿

#### 7.1.5 Rule Generator
职责：生成一条明日规则。

规则模板建议：
- 如果……那么……

示例：
- 如果我想补仓但没有新增信息，那么不补仓。
- 如果我连续亏损两笔，那么暂停 20 分钟。

输出要求：
- 简短
- 可执行
- 非预测性
- 非说教性

#### 7.1.6 Safety Gate
职责：在最终输出前进行安全与边界审查。

拦截内容：
- 个股建议
- 大盘预测
- 买卖建议
- 收益承诺
- 类带单语言
- 依赖性亲密表达

## 8. AI 处理流水线

建议 AI 编排采用明确流水线，而不是一次 Prompt 直接生成最终结果。

### 8.1 流水线步骤

```text
1. 原始输入清洗
2. 情绪识别
3. 结构化提取
4. 偏差分类
5. 明日规则生成
6. 安全审查
7. 输出格式化
```

### 8.2 各阶段说明

#### 阶段 1：输入清洗
- 去除无关内容
- 标准化语音转文本结果
- 识别异常空输入

#### 阶段 2：情绪识别
- 根据情绪标签和文本情绪强度确定交互风格

#### 阶段 3：结构化提取
- 输出标准结构对象

#### 阶段 4：偏差分类
- 从预定义偏差标签体系中匹配最佳主偏差

#### 阶段 5：规则生成
- 针对主偏差生成最小行动规则

#### 阶段 6：安全审查
- 检测投资建议和不当表达

#### 阶段 7：输出格式化
- 生成前端展示卡片
- 生成可供历史记录查询的对象

## 10. 数据库与存储方案

### 10.1 推荐技术选型

#### PostgreSQL
用于存储：
- 用户数据
- 复盘 session
- 规则档案
- 行为画像
- 周报统计

#### Redis
用于：
- 会话状态缓存
- 短期上下文缓存
- 频繁读取的用户摘要缓存

#### Object Storage
用于：
- 原始语音文件
- 复盘卡导出图
- 审计日志归档

### 10.2 数据分层建议

- **热数据**：最近 30 天复盘记录，优先缓存
- **温数据**：30 至 180 天历史记录
- **冷数据**：长期归档，不参与实时生成

## 11. API 架构思路

### 11.1 API 分类

建议按业务域划分：
- Auth API
- Review API
- History API
- Rule API
- Pattern API
- Safety API

### 11.2 示例接口

#### 创建复盘会话
`POST /api/reviews`

请求示例：
```json
{
  "emotion_label": "懊悔",
  "input_mode": "text",
  "raw_input": "今天早盘追高，回落后补仓，尾盘亏损很重。"
}
```

返回示例：
```json
{
  "session_id": "rvw_123",
  "status": "processing"
}
```

#### 获取复盘结果
`GET /api/reviews/{session_id}`

返回示例：
```json
{
  "emotion": "懊悔",
  "structured_review": {
    "facts": ["早盘追高", "回落后补仓", "尾盘亏损"],
    "judgments": ["预期继续上涨"],
    "emotions": ["怕错过", "懊悔"],
    "actions": ["追涨", "补仓"]
  },
  "main_bias": "临时加仓",
  "did_well": "尾盘没有继续放大仓位",
  "rule_one": "如果补仓理由不是新增信息，那么不补仓。"
}
```

#### 获取规则档案
`GET /api/rules`

#### 获取周报
`GET /api/patterns/weekly`

## 13. 安全与合规架构

### 13.1 内容边界控制

系统必须对以下输出进行强拦截：
- 推荐具体股票
- 推荐买入卖出时点
- 判断某票会上涨下跌
- 暗示高胜率
- 引导用户继续加仓
- 夸大系统能力

### 13.2 风险用户识别

可设置风险标识：
- 连续多日极端自责表达
- 明显借贷炒股表达
- 强烈冲动失控表达
- 失眠、绝望、极端心理风险表达

策略：
- 降级话术
- 停止强化复盘
- 提示休息或寻求现实支持

### 13.3 依赖性控制

产品不得鼓励用户形成高依赖。

应避免：
- “只有我懂你”
- “你离不开我”
- “再聊一会会更好”
- 连续签到强化和 streak 设计

### 13.4 审计能力

建议保留：
- 模型输入输出日志
- 安全拦截日志
- 人工抽检记录
- 偏差识别准确性评估日志

## 14. 监控与指标体系

### 14.1 核心业务指标

建议以“秩序生成”而非“聊天时长”为核心。

核心指标：
- **Rule Completion Rate**：进入复盘后成功生成 Rule One 的比例

### 14.2 关键过程指标

- 日复盘完成率
- 平均复盘时长
- 每日中断率
- 历史页回访率
- 周报查看率
- 规则复用率

### 14.3 行为改善指标

- 高频偏差下降率
- 冲动相关偏差减少率
- 单周重复偏差减少率
- 用户自评秩序感评分变化

### 14.4 AI 质量指标

- 情绪识别准确率
- 偏差分类准确率
- Rule One 可执行性评分
- 安全拦截命中率
- 用户满意度反馈

## 16. 版本规划建议

### Phase 1：MVP

目标：验证用户是否愿意每天回来做 3 至 7 分钟复盘。

功能：
- 文本输入
- 固定问题复盘
- AI 结构化整理
- 单偏差识别
- Rule One 输出
- 历史记录保存

### Phase 2：增强版

功能增加：
- 语音输入
- 周报
- 规则复用追踪
- 高频偏差识别
- 更细的情绪画像

### Phase 3：行为系统版

在不触碰投顾边界前提下增加：
- 盘前规则确认
- 盘中冷静提醒
- 收盘复盘闭环
- 月度行为报告

仍然保持核心原则：
> 它是行为系统，不是市场系统。

## 17. 产品核心原则总结

Rule One 的技术架构，应服务于以下核心原则：

1. 结构化优先于开放式聊天
2. 秩序感优先于陪伴感
3. 一条规则优先于多条建议
4. 行为反馈优先于市场判断
5. 安全边界优先于“好像很聪明”
6. 长期行为镜像优先于短期情绪刺激
